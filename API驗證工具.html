1 API URL 格式檢查

驗證要求：URL 必須符合以下組成要素：

  • 服務根網址 (Service Root URL, SRU)
  • 資源路徑 (resource path)
  • 查詢選項 (query options)
 符合 URL 結構完整
2 服務根網址的組成

驗證要求：SRU 必須符合以下組成要素：

  • scheme (協議，如 HTTPS)
  • host (主機)
  • port (導號)
  • URL 一律採用 HTTPS 區隔
 符合 使用 HTTPS 協議
3 提供「Landing Page」 必須有一個起始頁面，連結 API 資源並提供相關說明。 不符合 缺少標準 landing page，僅有 API 端點
4 API 定義文件格式 API 定義文件必須採用 JSON 或 YAML 格式。 符合 提供 JSON 格式文件
5 特定查詢功能支援

API 必須支援以下查詢參數：

  • bbox：限制回傳資料的空間範圍 (可為二維或三維坐標值)
  • datetime：限制回傳資料的時間範圍，可為特定時間點或區間
  • limit：限制回傳資料的筆數
 部分符合 缺少 datetime 參數支援

Landing Page 必須包含「collection」連結，並提供以下資訊：

  • 資料特徵的識別碼 (如唯一 ID)
  • 坐標參考系統資訊
  • 名稱與描述 (選擇性)
  • 空間或時間範圍 (選擇性)
 不符合 缺少 collection 連結

3. 缺少 Landing Page

問題描述：API 服務缺少標準的 landing page，無法提供 API 資源的整體說明和導覽。

建議解決方案：

  • 建立符合規範的 landing page，包含 API 基本資訊
  • 在 landing page 中提供所有可用資源的連結
  • 範例符合的 landing page: https://api.example.com/

5. 部分查詢功能缺失

問題描述：API 缺少 datetime 查詢參數支援，無法按時間範圍篩選資料。

建議解決方案：

  • 實作 datetime 查詢參數功能
  • 支援 ISO 8601 時間格式
  • 範例符合的查詢: ?datetime=2023-01-01T00:00:00Z/2023-12-31T23:59:59Z

7. 缺少 Collection 連結

問題描述：Landing page 中缺少 collection 資源的連結和基本描述。

建議解決方案：

  • 在 landing page 中添加 collections 連結
  • 為每個 collection 提供基本描述
  • 範例符合的結構: 
    {
  "collections": [
    {
      "id": "housing",
      "title": "住宅資料",
      "description": "全台住宅相關資料集"
    }
  ]
}
